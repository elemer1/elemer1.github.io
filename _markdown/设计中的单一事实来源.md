# The Single Source of Truth in Design: From Pixels to Tokens to the Agentic Codebase

> *"Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."* — Andy Hunt and Dave Thomas, *The Pragmatic Programmer* (1999), the DRY principle

The phrase **Single Source of Truth (SSOT)** is usually introduced in product teams as a collaboration nicety: a shared place everyone agrees to trust, so that designers and engineers stop arguing about what the button is supposed to look like. That framing is too small. A source of truth is an entropy-control mechanism. It is the answer to a precise question — *when two representations of the same thing disagree, which one is right?* — and the cost of getting that answer wrong compounds with every person, platform, and release that depends on it.

This essay makes one argument, and it is deliberately narrower than the triumphant version you often hear. The single source of truth in design has never been a *medium* (a Photoshop file, a Sketch library, a Figma project). It is a *contract* about which representation is authoritative for which concern. What the era of large language models and autonomous agents is doing is not crowning a winning tool. It is collapsing the authoritative representation *downward*, out of the hand-maintained visual canvas and into a machine-readable model of structured data that humans and machines can read from the same place. Code is reclaiming primacy as the physical substrate of that truth. It is not becoming the universal interface, and the tools that visualize design are not dying. They are repositioning to sit on top of the same substrate.

To defend that claim properly, the concept has to be taken apart along four seams: its genealogy in information systems, its internal anatomy in a modern design system, the historical migration of control across design tools, and the fault line between design intent and physical reality that the agentic shift is finally forcing into the open.

## I. What "Single Source of Truth" Actually Means, and Where It Came From

The idea did not originate in design. It was borrowed, decades late, from the discipline that had already paid for the mistake.

In 1970, E. F. Codd published the relational model of data, and with it the practice of *normalization*: structuring a database so that every fact lives in exactly one place, eliminating the redundant copies that drift out of sync the moment one of them is updated and the others are not. Normalization is the SSOT, expressed mathematically. Nearly thirty years later, Andy Hunt and Dave Thomas gave the same idea its most quoted form in *The Pragmatic Programmer* (1999) as the DRY principle, *Don't Repeat Yourself*: "every piece of knowledge must have a single, unambiguous, authoritative representation within a system." Read that sentence again. It is not a slogan about typing less code. It is the literal definition of a single source of truth, and it is the standard against which every design workflow in this essay should be judged.

If DRY explains what a source of truth *is*, Melvin Conway explained in 1968 *why it keeps fracturing*. Conway's Law observes that organizations are "constrained to produce designs which are copies of the communication structures of these organizations." A company with a design team and a separate engineering team, talking across a handoff, will produce two representations of its interface, because the org chart has two boxes. The split source of truth is not a tooling failure. It is the org chart, made visible in the artifacts.

This genealogy matters because it reframes the whole problem. Software engineering solved version control, normalization, and DRY for itself decades ago, while visual design lagged. The lag had a simple cause: code is text, and text is data, so the machinery of authoritative representation (diffs, merges, single sources) attaches to it naturally. Design's artifacts were pictures, and a picture is not data. You cannot diff two JPEGs in any meaningful sense, cannot merge them, cannot ask one of them what its primary color "means." The history that follows is largely the history of design slowly turning its pictures into data, until the data became indistinguishable from code, at which point the original question (which representation is authoritative?) returns with full force.

## II. The Anatomy of a Design Source of Truth

A mature design system encodes its truth in three strata of increasing abstraction. The strata are real and named; this is not a taxonomy invented for the essay.

**Design tokens.** The smallest unit. A token replaces a hard-coded physical value with a named, semantic variable: `#FF0000` becomes `color-action-danger`, `16px` becomes `spacing-md`. The term was coined by Jina Anne while she led design on the Salesforce Lightning Design System around 2014, with the canonical definition that tokens are "the visual design atoms of the design system — specifically, they are named entities that store visual design attributes," used "in place of hard-coded values in order to maintain a scalable and consistent visual system." The decisive property is platform-agnosticism. A token is authored once, typically as JSON, and *compiled* into whatever each platform consumes: CSS custom properties for the web, Swift constants for iOS, XML for Android. The tooling that performs that compilation is itself notable history: Salesforce built an open-source transformer called Theo (first written by Sönke Rohde), and Amazon's Danny Banks open-sourced Style Dictionary in 2017, which became the dominant tool for turning tokens into production code. The W3C's Design Tokens Community Group (DTCG), formed in 2019, has since been standardizing the JSON format itself.

Sit with one implication, because the entire argument of this essay turns on it: *the most truth-like layer of design is already structured data*. The part of design that most deserves the word "truth" — the canonical record of what color, what spacing, what type scale — was always code-shaped. It was a JSON file pretending to be a swatch.

The mature taxonomy splits tokens into three tiers, a structure used by Salesforce, Google's Material Design 3, and the DTCG specification alike: *primitive* tokens (`blue-500: #3B82F6`, raw values with no meaning), *semantic* tokens (`color-action-primary`, which aliases a primitive and carries intent), and *component* tokens (`button-background-color`, scoped to a specific element). Each tier references the one below, so a brand change propagates through aliases rather than through find-and-replace. This is normalization, applied to color.

**Components.** Built on top of tokens, a component library constrains the visual form and structure of reusable elements across their lifecycle states: *default*, *hover*, *focus*, *disabled*. The conceptual scaffolding here is Brad Frost's *Atomic Design* (2016), which borrowed from chemistry to describe interfaces as atoms composed into molecules, molecules into organisms, and so on up to pages. Every serious design system instantiates some version of this: Material (Google), Polaris (Shopify), Carbon (IBM), Fluent (Microsoft), Primer (GitHub). A component is best understood as a small state machine whose visual output is determined by its props and its current state, which is precisely how it will be implemented in React or SwiftUI. The design representation and the code representation are already describing the same finite-state object in two notations.


**Guidelines and interaction specifications.** The top stratum governs composition and behavior: where the primary button sits relative to the cancel button in a dialog, how a panel eases open, which motion curve signals what. This layer is the least data-like and the most prose-like, which is exactly why it has resisted automation longest and why it will be the last thing to migrate.

The shape of the stack is the point. The bottom is pure data, the middle is a state machine, and only the top is genuinely soft. Design's truth was always mostly structured; the tools simply hid that fact behind a canvas.

## III. A Tool History Is a History of Who Holds the Truth

The succession of design tools reads on the surface as a story of better software. Underneath, it is a story about where the authoritative representation lives and who is forced to translate around it.

**The bitmap era.** Under Photoshop and Fireworks, there was no shared source of truth, because design assets were static rasters. A design "spec" was a flattened picture, and translating it into a running interface meant an engineer measuring pixels by eye and slicing exported images. Every value crossed the design-to-code boundary as a human-read-and-retyped number, which is to say through the noisiest channel available. The truth, such as it was, lived in the engineer's reconstruction, not in the picture.

**The vector-symbol era.** Sketch (2010) introduced symbols and, later, shared Libraries, letting teams modularize and reuse interface elements. For the first time the design file itself carried structure rather than flattened pixels. The structure remained local and static, though: a file on a designer's machine, exported and emailed, with no live link to anything downstream.

**The cloud-relational era.** Figma, founded in 2012 by Dylan Field and Evan Wallace and opened to the public in 2016, abandoned the canvas-as-file metaphor for a browser-native, multiplayer document that behaves like a concurrent relational database with a pseudo-code logic layer. By binding together Variables, Variants, and Auto Layout (an elastic layout engine that mirrors the web's box model), Figma manufactured an industry consensus: the Figma file is the standard, and engineers reverse-engineer production code from it. The dominance was not marginal. When Adobe agreed to acquire Figma for $20 billion in 2022, regulators in the European Union and the United Kingdom blocked the deal on the grounds that removing Figma would harm competition and innovation in design software; the deal collapsed in December 2023, with Adobe paying a $1 billion termination fee. Figma then went public on the New York Stock Exchange in July 2025 under the ticker FIG, and the stock rose roughly 275% on its debut, briefly valuing the company near $68 billion. These are not the numbers of a tool in retreat. They are the backdrop against which any claim about code "reclaiming" the truth must be made with care.

**The handoff interregnum.** One detail of this period is easy to skip and analytically decisive. An entire category of tools existed for years whose only purpose was to bridge design and code: Zeplin and InVision Inspect for generating specs and assets, Abstract for putting design files under Git-style version control. A thriving market for *translation tools* is direct evidence that the source of truth was split in two. Nobody builds a bridge across a river that is not there.

## IV. The Fault Line: Intent Versus Reality

Treating a visual design tool as the absolute arbiter introduces a problem that is, in the strict sense, philosophical. A design file extrapolates an *intent*: an approximation of what the product should be. The code running on a user's device is the *reality*: the only physical instantiation of the product that actually exists. These are different kinds of thing, and the entire design profession spent two decades pretending they were the same.

The cleanest way to name the gap is Korzybski's: the map is not the territory. A Figma frame is a map of an interface. The deployed build is the territory. One can be exquisitely detailed and still be a map. Borges pushed the idea to its limit in "On Exactitude in Science" (1946), imagining cartographers who draw a map so faithful it is the same size as the empire and coincides with it point for point, whereupon it becomes useless and is left to rot in the desert. A design mockup detailed enough to be perfectly faithful to the running code would simply *be* the code; short of that, it is a representation, and representations drift from what they represent. Plato made the older version of the point: the design is the Form, an ideal that no physical instance fully realizes, and the build is the imperfect particular that ships.


This abstract gap produces two concrete, recurring failures.

**The translation penalty.** Joel Spolsky's Law of Leaky Abstractions (2002) holds that "all non-trivial abstractions, to some degree, are leaky." Figma's primitives — deeply nested component properties, variant matrices, auto-layout constraints — are proprietary, which means an engineer must spend real cognitive budget translating them into React, Vue, or native code. Cross-medium translation is never lossless. Something is always approximated, and the approximation is paid for in bugs and in time.

**Source collapse.** The more damaging failure is one of direction. In fast iteration, engineers routinely adjust styles or state logic directly in the codebase to fix a production bug or claw back performance. Unless those changes are back-ported into the design file — an expensive, manual, reverse-alignment process that almost never fully happens — the design representation silently decays into an idealized blueprint detached from what users actually see, while the repository becomes the wild, organic, lived truth. The map and the territory diverge, and everyone quietly learns to trust the territory and ignore the map. This is the empirically familiar phenomenon of design-code drift, and it is the symptom that exposes which representation was ever really authoritative.

There is an old principle from architecture that names the honest response. *Truth to materials*, the modernist and Arts-and-Crafts doctrine, holds that a material should be used according to its own nature and not disguised as another: do not paint pine to look like oak, do not mold concrete to imitate stone. Applied here: if the material that ships to users is a web or app interface rendered from code, then the honest place to declare and hold the truth is that code, not a picture of it.

## V. The Return to Code Began Before AI

It would be a mistake — and the mistake the triumphalist version of this argument makes — to credit large language models with discovering that the truth belongs in code. The gravitational pull toward a code-resident source of truth was well underway before agents could write a line, driven by engineers solving their own pain. The honest claim is that AI did not invent this movement. It removed the last friction from it.

The evidence is a decade of tooling that quietly relocated authority into the codebase:

Style Dictionary and Theo made design tokens *primarily* a text artifact. The canonical color was a value in a JSON file, transformed into every platform's format at build time. The truth lived in a file an engineer could diff and a build system could compile, not on a canvas.

Storybook turned components into a live, code-resident gallery, documenting and testing each component as the running thing it actually is, rather than as a picture of the thing. The "real" component library became the one in the repository.


Tailwind CSS collapsed the design system into a configuration file. A team's spacing scale, color palette, and type ramp became `tailwind.config`, a code object that the styling system reads directly. The design system *was* the config.

shadcn/ui, which arrived in 2023 and became the year's most-starred new JavaScript project, pushed the logic to its conclusion with an ownership model that inverts the traditional library. Instead of installing an opaque package and inheriting its decisions, you run a command and the component's actual source code is copied into your repository. You own it; you edit the file. There is no black box and no vendor lock-in, only readable code you control. By design, shadcn/ui located the source of truth *in the user's own codebase* before AI agents were a serious consideration.


A code-first source of truth, in other words, was already the direction of travel. What changed is that a new kind of reader arrived.

## VI. The Agentic Inflection

Large language models tilt the field toward code for a reason rooted in what they are. Their competence is built on the open corpus of the internet: HTML, CSS, JavaScript, and the abstract syntax trees of real programs, all of it linear, structured text. A model natively understands a DOM tree and the box model because it has read millions of them. It reads text and code fluently. It does not natively read a proprietary binary node tree, because that representation is closed, non-textual, and absent from its training data. Confronted with a design system expressed as code versus the same system expressed as a tool's private file format, an agent finds the code legible and the file format opaque.

This is why the new generation of generative interface tools is code-first by construction. Vercel's v0, along with Bolt, Lovable, Cursor, and Claude's own artifacts, turn a prompt or an image into working React and Tailwind. The most telling fact in the whole landscape is *why* these tools converged on shadcn/ui as their default component vocabulary: the visible, owned source is easier for a model to read and modify than an opaque library abstraction. The agent can see the implementation, so it can change it reliably. An interface system optimized for machine reading turned out to be an interface system made of plain, owned code.

The distribution mechanism is evolving to match. shadcn's registry model distributes components as source plus JSON metadata rather than as installed packages, complete with "Open in v0" buttons and MCP endpoints, so a design system can be delivered in a form that humans copy and agents fetch over a protocol. The artifact and its machine interface are becoming the same thing.

## VII. Steelman: Why "Code Monism" Overshoots

A benchmark argument has to defeat its strongest opponent, not its weakest. Here is the case against the thesis, made at full strength, in four moves. The first one alone is enough to retire the cruder version of "code wins."

**Figma is not inert; it is racing to own the interface to the code truth.** The claim that AI cannot parse Figma's node tree was defensible in 2023 and is now false, because Figma built the bridge itself. Figma shipped Dev Mode in 2023, and in 2025 a Dev Mode MCP server that exposes a file's structured contents — hierarchy, layout rules, token values, component properties — to agentic coding tools (Cursor, GitHub Copilot, Windsurf, Claude Code) over the Model Context Protocol, the open standard that lets AI read software directly instead of guessing from a screenshot. Code Connect maps a design component to the exact code component and file path in the repository, so the agent is told which component to use rather than inventing one; Figma's own documentation calls this "a shared source of truth for both the design and code elements." And Figma Make (2025) generates working React and Tailwind from a prompt, running on Anthropic's Claude. The irony is exact: the design tool the thesis predicts will be displaced by AI is itself an AI code generator built on the same model family doing the displacing. The realistic outcome is not Figma's death. It is Figma as the structured, human-facing front end to a source of truth that physically lives in code.

**Design's divergent phase resists premature code.** The generative front end of design — exploration, divergent options, research, motion studies, brand work — derives its value from *not* being committed to an implementation. A fluid, low-stakes medium is where bad ideas die cheaply and unexpected ones survive. Code is a convergence medium; it is excellent for production and poor for the fuzzy phase where the question is still "what should this even be." Forcing the exploratory phase into code risks premature commitment, a well-documented failure mode in which the cost of having built something becomes an argument against abandoning it.

**Code is not a universal interface, and a single source of truth has to serve the whole organization.** Most of a company cannot read a DOM tree or a pull-request diff: product managers, marketers, executives, content strategists, legal reviewers. A medium that only engineers can edit is not a single source of truth for the organization; it is a single source of truth for engineering, which is precisely the kind of siloed authority Conway's Law warns about. A real SSOT requires role-appropriate *views* onto the underlying truth, and a visual canvas or a live preview is exactly such a view. "Everyone reads the code" is an engineer's fantasy about an organization made entirely of engineers.

**Natural language to code reintroduces a lossy translation; it does not abolish one.** The thesis indicts Figma's design-to-code gap as the source of information loss. But prompt-to-code has its own translation penalty, and it may be worse, because natural language is more ambiguous than a vector file. Models hallucinate, misread intent, and — as Figma's own MCP documentation notes — will fabricate a new component from a screenshot when they cannot find a match for it. The lossy intermediary did not vanish in the agentic era. It changed costume, moving from a visual medium to a linguistic one.

## VIII. The Synthesis: One Model, Many Views

The way out of the false binary ("Figma versus the codebase," "design tool versus code") is to notice that both sides have been converging on the same thing from opposite directions, and that the thing is neither a tool nor a medium.

The code side has been moving *up* toward distributable, machine-readable structured truth: design tokens authored as DTCG-standard JSON, shadcn-style component registries that ship source plus metadata, typed component contracts that double as documentation. The design-tool side has been moving *down* toward emitting code and exposing structure: Figma's Variables, Dev Mode, MCP server, Code Connect, and Make. They are meeting in the middle, and the meeting point is a **structured model** — tokens, component contracts, registries — expressed as data that is authoritative for both design and code at once.

This reframes the source of truth correctly. It was never a *medium*; it is a *model*. The agentic era is collapsing that model *out of* the hand-maintained mockup and the hand-typed stylesheet and *into* a structured, machine-readable layer onto which every participant attaches a view suited to their role: a designer attaches a visual canvas, an engineer attaches a code editor, a product manager or executive attaches a live preview, and an AI agent attaches over a protocol such as MCP. One model, many synchronized views. The map and the territory finally query the same database.

Two of the lenses worth holding here sharpen the conclusion.

Invert the question, in the manner Charlie Munger borrowed from Jacobi: instead of asking "what is the ideal source of truth," ask "what produces design-code drift and collaboration failure?" The answer is uncomfortable and clarifying. Every *translation* between representations and every *reconciliation* of parallel copies is a site of entropy. It follows that the goal is not to crown a winning representation. It is to eliminate parallel representations entirely: one authoritative model, many views read from it, the smallest possible number of translations between them. That is, exactly, the DRY principle from 1999 — one unambiguous, authoritative representation — restored to the discipline that took longest to adopt it.

Donella Meadows' framework on where to intervene in a system explains why this is a deep change rather than a tooling preference. A source of truth is an intervention on *information flows* (the sixth of her twelve leverage points, concerning who has access to what information) and on the *rules of the system* (the fifth). Relocating authority from "the picture is real" to "the structured model is real, and code is its substrate" is an intervention on the system's *paradigm* (the second and highest-leverage point short of transcending paradigms): it changes the shared, mostly unspoken assumption about which artifact is the actual product. Buying a better design tool is a low-leverage parameter change. Changing where the truth is allowed to live is a paradigm change, and paradigm changes are where the leverage is.

So the honest, defensible form of the original thesis is this. Code is reclaiming primacy as the *physical substrate* of design truth, for the unglamorous reason that code is what ships and code is what machines read. Code is not becoming the *universal interface*, because most of the people and processes that depend on the truth cannot and should not read it directly. The era of maintaining two hand-aligned realities — a mockup and a build, drifting apart between every release — is ending. The era of one structured model behind many interfaces is beginning. Asking whether Figma or the codebase wins is the wrong question. The winner is the structured layer between them, and the organizations that win are the ones that minimize the number of times anyone or anything has to translate across it.

## Coda

A single source of truth is not, in the end, a convenience for design teams. It is the precondition for control. You cannot run a closed feedback loop over a system whose state is recorded in two places that disagree, because the loop never knows which signal to trust, and a control system fed contradictory state does not converge — it oscillates. For three decades, digital product teams ran exactly that broken loop: a design representation and a code representation, each claiming to be real, reconciled by hand and never quite reconciled. What the agentic era offers is not faster pictures. It is the chance to close the loop at last, by letting the map and the territory finally be read from the same source.

- Design tokens: coined by Jina Anne at Salesforce on the Lightning Design System, c. 2014; canonical definition and the three-tier primitive/semantic/component taxonomy (also used by Google Material Design 3 and the W3C DTCG). Tooling: Theo (Salesforce, Sönke Rohde) and Style Dictionary (Danny Banks, Amazon, open-sourced 2017). Standardization: W3C Design Tokens Community Group, formed 2019.
- Figma history and dominance: founded 2012 (Dylan Field, Evan Wallace), public 2016; Variables and Dev Mode launched 2023. Adobe's $20B acquisition agreed 2022, terminated December 2023 under EU/UK regulatory pressure with a $1B termination fee; Figma IPO on NYSE (ticker FIG), July 2025, ~275% first-day rise, briefly ~$68B market capitalization.
- Figma's AI and code bridges: Dev Mode MCP server (beta 2025), exposing structured file data to Cursor, GitHub Copilot, Windsurf, and Claude Code over the Model Context Protocol; Code Connect, mapping design components to code component/file paths and described by Figma as "a shared source of truth for both the design and code elements"; Figma Make (announced Config 2025, GA July 2025), prompt-to-React/Tailwind, running on Anthropic's Claude.
- Code-first lineage: Style Dictionary, Storybook, Tailwind CSS config-as-design-system; shadcn/ui (2023), the copy/ownership model and component registry, adopted as the default vocabulary by v0 (Vercel), Bolt, Lovable, and Cursor because owned source code is more legible to AI models than opaque library abstractions.
